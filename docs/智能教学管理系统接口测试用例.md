### 10.3 AI 对话 `POST /api/chat`

**说明**：该接口使用 SSE（Server-Sent Events）流式返回，`Content-Type` 为 `text/event-stream`。响应是逐 token 输出的文本流，不是 JSON 格式。

**测试场景1：单轮对话**
- **请求URL**：`POST http://localhost:8080/api/chat`
- **请求头**：`Content-Type: application/json`
- **请求体**：
  ```json
  {
    "question": "你好，请介绍一下你自己",
    "sessionId": "a1b2c3d4e5f6g7h8"
  }
  ```
- **预期响应**（SSE 流式文本，逐 token 输出）：
  ```
  data: 你好
  data: ！我是
  data: 教学管理
  data: 系统的
  data: AI智能
  data: 助理
  data: ...
  data: [DONE]
  ```

**测试场景2：多轮对话（带上下文）**
- **前置条件**：先在当前 session 中发送过问题"我叫张三"
- **请求体**：
  ```json
  {
    "question": "我叫什么名字？",
    "sessionId": "a1b2c3d4e5f6g7h8"
  }
  ```
- **预期响应**：AI 能根据历史消息回答"你叫张三"

**测试场景3：sessionId 不存在（新会话）**
- **请求体**：
  ```json
  {
    "question": "帮我推荐一门课程",
    "sessionId": "nonexistent_session_id"
  }
  ```
- **预期响应**：正常回答，但无历史上下文（等同于新会话的第一条消息）

**测试场景4：空问题**
- **请求体**：
  ```json
  {
    "question": "",
    "sessionId": "a1b2c3d4e5f6g7h8"
  }
  ```
- **预期响应**：AI 可能返回空内容或提示输入问题


### 10.4 AI 模块接口汇总

| 接口方法 | 接口路径            | 功能描述                           | 返回类型                |
| -------- | ------------------- | ---------------------------------- | ----------------------- |
| POST     | `/api/session`      | 创建AI会话，返回sessionId和示例问题 | JSON (`SessionVO`)      |
| GET      | `/api/session/hot`  | 获取随机热门示例问题               | JSON (`List<Example>`)  |
| POST     | `/api/chat`         | AI 对话（SSE流式返回）             | SSE (`text/event-stream`) |

### 10.5 完整测试流程

1. **创建会话** → 调用 `POST /api/session` 获取 `sessionId`
2. **获取热门问题** → 调用 `GET /api/session/hot` 查看推荐问题列表
3. **发起对话** → 调用 `POST /api/chat` 携带 `sessionId` 和 `question`，通过 SSE 接收流式回答
4. **多轮对话** → 使用相同的 `sessionId` 继续提问，验证 AI 能记住上下文
5. **新会话** → 再次调用 `POST /api/session` 获取新的 `sessionId`，开启全新对话


## 测试注意事项

1. **身份验证**：所有接口都需要在请求头中携带 `Authorization: Bearer {token}`
2. **权限控制**：不同角色拥有不同的接口访问权限
3. **请求格式**：所有POST和PUT请求需要设置 `Content-Type: application/json`
4. **分页参数**：所有列表接口支持 `page` 和 `pageSize` 参数
5. **错误处理**：接口返回统一的错误格式，包含code、message和data字段
6. **数据准备**：测试前需要先创建测试数据，如用户、课程等
7. **测试顺序**：建议按照依赖关系顺序测试，如先测试认证模块，再测试其他模块

以上测试用例覆盖了系统的所有接口，可以根据实际情况进行调整和扩展。
